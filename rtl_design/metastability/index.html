Physics

The Bistable Energy Landscape

A D flip-flop is a bistable circuit — it has exactly two stable energy minima: State 0 and State 1. But between them sits an unstable equilibrium — the metastable point. Any tiny perturbation (thermal noise, a passing cosmic ray) will eventually push the state to one side. The problem: resolution time is theoretically unbounded.

Fig 1 — Bistable potential well. The ball represents the flip-flop state. Setup/hold violation kicks it to the metastable peak — thermal noise eventually pushes it to State 0 or State 1.

Timing

The Setup and Hold Window — The Forbidden Sampling Zone

Every flip-flop has a setup time (tsu) and hold time (th) requirement relative to the clock edge. If DATA transitions anywhere inside this window, the FF cannot guarantee a valid output.

Danger

Why Metastability Destroys Chips — Even After It Resolves

The most dangerous aspect is not the mid-rail voltage itself — it is what happens to the fanout before resolution.

⚡ Fanout Disagreement

FF1_Q at 0.55V feeds three downstream gates. Gate A (NAND) interprets it as '1'. Gate B (NOR) interprets it as '0'. Both are correct — for their own thresholds. But now your logic is in an impossible state: A=1 AND B=0 from the same wire.

🔥 Crowbar Current

When a CMOS input sees mid-rail voltage (~VDD/2), both the PMOS and NMOS transistors in the gate are partially on. This creates a direct path from VDD to GND — crowbar current. On a chip with millions of gates, this can cause localized heating and electromigration.

🔒 State Machine Lockup

If a metastable value reaches an FSM's state register, the FSM can enter a state that no valid encoding represents. The machine may deadlock in a loop with no exit condition. This is the bug that famously caused the Therac-25 radiation overdose incidents — race conditions creating impossible states.

🛰 System Failure

The SOHO satellite (1998) experienced a loss of attitude control traced to unprotected CDC crossings. The Mars Climate Orbiter (1999) loss involved unit conversion errors compounded by timing issues. Metastability in safety-critical systems is not theoretical — it has caused real disasters.

Solution

The 2-FF Synchronizer — Industry Standard Fix

The 2-FF synchronizer is the simplest and most common solution. For high-reliability designs (aerospace, automotive ISO 26262), use 3 stages. Never use a latch as a synchronizer — latches have no setup/hold guarantee.

// 2-FF synchronizer — correct implementation
module sync_2ff #(parameter W = 1) (
  input  logic        clk_dst,   // destination clock
  input  logic        rst_n,
  input  logic [W-1:0] d_async,   // async input from src domain
  output logic [W-1:0] d_sync     // clean output in dst domain
);
  // IMPORTANT: both FFs must be placed by the tool in the same
  // physical location (max_delay / dont_touch constraints needed)
  (* dont_touch = "true" *) logic [W-1:0] ff1_q, ff2_q;

  always_ff @(posedge clk_dst or negedge rst_n) begin
    if (!rst_n) begin
      ff1_q <= '0;
      ff2_q <= '0;
    end else begin
      ff1_q <= d_async;   // may go metastable — that's OK
      ff2_q <= ff1_q;     // samples clean ff1 one period later
    end
  end

  assign d_sync = ff2_q;
endmodule

// SDC constraint — critical: give FF1→FF2 path a max_delay
// equal to one clock period so the tool knows it's a sync path
set_max_delay -datapath_only 2.0 \
  -from [get_cells sync_inst/ff1_q_reg*] \
  -to   [get_cells sync_inst/ff2_q_reg*]
SDC constraint is mandatory. Without set_max_delay -datapath_only, static timing analysis will apply the normal clock-to-clock multi-cycle check on the FF1→FF2 path and may incorrectly try to optimize it, potentially placing FF1 and FF2 far apart — defeating the synchronizer's purpose.
// Gray-coded counter for multi-bit CDC (e.g., async FIFO pointer)
// Rule: only ONE bit changes per clock — so even if one bit is
// sampled metastably, the worst case is off-by-one, not garbage
function automatic logic [W-1:0] bin2gray (logic [W-1:0] bin);
  return bin ^ (bin >> 1);
endfunction

// NEVER synchronize a binary counter directly across a CDC boundary
// Multiple bits change simultaneously → multiple sync failures possible
FAQ

Frequently Asked Questions

Metastability occurs when a flip-flop samples its data input during the setup+hold window around the clock edge. The FF enters an analog state at approximately VDD/2 — neither a valid logic '0' nor '1'. It will eventually resolve, but the resolution time is random and potentially unbounded. If downstream logic samples the output before it resolves, it can interpret the mid-rail voltage as different values at different gates, corrupting the design state.

Any data transition that occurs within t_setup before the clock edge OR within t_hold after the clock edge. This is most common at clock domain crossings (CDC) because two independent clocks have no fixed phase relationship — given enough time, the data edge will inevitably land inside the forbidden window. It can also happen with asynchronous resets, button inputs, or any signal not synchronized to the sampling clock.

Both FFs are clocked by the destination clock. FF1 samples the asynchronous input and may go metastable. The metastable state decays exponentially with time constant τ. By the time FF2 samples FF1 (one full clock period later), FF1 has had time to resolve. At 100 MHz with τ = 30 ps, after one 10 ns period the probability of FF1 still being metastable is e^(−10000/30) ≈ 10^(−145) — far smaller than any engineering concern.

No. Static Timing Analysis checks timing within a clock domain and with proper constraints. CDC crossings without set_max_delay or set_false_path constraints appear as timing violations, but STA cannot detect metastability probability or check that synchronizers are architecturally correct. You need dedicated CDC analysis tools (Synopsys SpyGlass CDC, Cadence JasperGold CDC) to verify that every CDC path has a proper synchronizer and that Gray coding is used for multi-bit signals.

A 2-FF synchronizer works for a single bit because only one thing can be metastable. For an N-bit binary counter, multiple bits change simultaneously on each increment. The destination domain may sample some bits in the old state and some in the new state — creating an intermediate value that was never a valid count. The solution is Gray coding (only 1 bit changes per step) for counters, or handshake protocols (req/ack across FIFOs) for arbitrary data buses.
