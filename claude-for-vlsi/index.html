Live conversation example
I have a 2-FF synchronizer in my CDC path but still see metastability failures in simulation. Why?
A 2-FF synchronizer reduces metastability probability but cannot eliminate it. The mean time between failures (MTBF) is:

MTBF = exp(Tw/τ) / (f_clk × f_data × C1)

If you're still seeing simulation failures, the likely causes are:
1. Resolution time τ is too short for your process corner
2. The data path violates the sampling window assumption
3. You're using $random stimulus that toggles faster than MTBF assumptions allow...

Prompt 1 — Generate a parameterized FIFO
You are an expert RTL design engineer. Write a parameterized synchronous FIFO in Verilog with the following spec:
• DATA_WIDTH and DEPTH parameters
• Synchronous reset (active high)
• Full and empty flags
• almost_full when depth − 2 entries are used
• Use non-blocking assignments (<=) in all sequential blocks
• Target: Xilinx UltraScale BRAM inference
Include a brief comment explaining the ptr gray coding approach for FIFO depth tracking.
Claude Response (excerpt)
Here's a synthesizable parameterized FIFO targeting Xilinx UltraScale BRAM: 
module sync_fifo #(
  parameter DATA_WIDTH = 8,
  parameter DEPTH      = 16,
  parameter ADDR_W     = $clog2(DEPTH)
)(
  input  wire                  clk, rst,
  input  wire                  wr_en, rd_en,
  input  wire [DATA_WIDTH-1:0] din,
  output reg  [DATA_WIDTH-1:0] dout,
  output wire                  full, empty, almost_full
);
  (* ram_style = "block" *)
  reg [DATA_WIDTH-1:0] mem [0:DEPTH-1];
  reg [ADDR_W:0] wr_ptr, rd_ptr;  // extra bit for full/empty distinction

  always @(posedge clk) begin
    if (rst) begin
      wr_ptr <= 0;  rd_ptr <= 0;
    end else begin
      if (wr_en && !full)  begin mem[wr_ptr[ADDR_W-1:0]] <= din; wr_ptr <= wr_ptr + 1; end
      if (rd_en && !empty) begin dout <= mem[rd_ptr[ADDR_W-1:0]];  rd_ptr <= rd_ptr + 1; end
    end
  end

  assign empty       = (wr_ptr == rd_ptr);
  assign full        = (wr_ptr[ADDR_W] != rd_ptr[ADDR_W]) &&
                       (wr_ptr[ADDR_W-1:0] == rd_ptr[ADDR_W-1:0]);
  assign almost_full = (wr_ptr - rd_ptr) >= (DEPTH - 2);
endmodule
The extra pointer bit technique avoids the ambiguity between full (wr==rd) and empty — they differ only in the MSB.

Prompt 4 — Generate SVA assertions
Write SystemVerilog concurrent assertions (SVA) for an AXI4-Lite slave interface. Cover these properties:
1. awvalid must not deassert before awready is seen
2. wvalid must not deassert before wready is seen
3. bvalid must be asserted exactly 1–4 cycles after the write handshake
4. araddr must be stable while arvalid is asserted and arready is low
5. No outstanding read request during a write (exclusive access)

Use clocking block @(posedge clk). Include assume properties for input stimulus and assert properties for DUT behavior. Add $error messages.
Claude Response (excerpt)
// AXI4-Lite SVA Assertions
module axi_lite_sva #(parameter DW=32)(
  input logic clk, rst_n,
  // Write address channel
  input logic awvalid, awready,
  input logic [31:0] awaddr,
  // Write data channel
  input logic wvalid, wready,
  // Write response
  input logic bvalid, bready,
  // Read address
  input logic arvalid, arready,
  input logic [31:0] araddr
);
  default clocking cb @(posedge clk); endclocking
  default disable iff (!rst_n);

  // P1: awvalid must not deassert before awready
  p_awvalid_stable: assert property (
    awvalid && !awready |=> awvalid
  ) else $error("AXI: awvalid dropped before awready");

  // P3: bvalid within 1–4 cycles of write handshake
  p_bvalid_latency: assert property (
    (awvalid && awready) ##0 (wvalid && wready)
    |-> ##[1:4] bvalid
  ) else $error("AXI: bvalid not seen within 4 cycles of write");

  // P4: araddr stable while arvalid high and arready low
  p_araddr_stable: assert property (
    arvalid && !arready |=> $stable(araddr)
  ) else $error("AXI: araddr changed while arvalid high");

Claude Response (excerpt)
Looking at your timing report, the setup violation (slack = −0.23ns) is caused by a long combinational path through 5 cascaded mux cells in the address decoder, combined with a high-fanout net on the select signal.

Critical path analysis:
• Launch FF → U_DEC/MUX_0 (0.18ns cell delay + 0.34ns wire)
• MUX chain: 5× 0.21ns = 1.05ns
• Capture FF setup: 0.08ns

Fix strategies (ranked):
1. Logic restructuring (best ROI): Replace the 5-deep mux chain with a priority encoder + one-hot decode — reduces to 2 logic levels
2. Placement hint: set_max_delay -datapath_only 2.8 [get_pins U_DEC/...] allows the placer to co-locate the cells
3. Buffer insertion: Add a repeater on the high-fanout select net — reduces skew from 0.21ns to ~0.08ns

Prompt 10 — Floorplan strategy
I'm floorplanning a 7nm mobile SoC with these blocks: CPU cluster (4× A78, 2mm²), GPU (1.5mm²), NPU (0.8mm²), DDR PHY, USB PHY, PCIe PHY, SRAM banks (32× 256KB). Die size is 8mm × 8mm, 4 metal power rails.

Suggest a floorplan strategy covering:
1. Block placement priority and adjacency reasoning
2. Power domain partitioning (always-on vs power-gated)
3. PHY placement near die edge rationale
4. SRAM placement for minimum latency to CPU
5. Clock tree root placement
Claude Response (excerpt)
Floorplan strategy for your 7nm SoC:

1. Block placement priority:
Start with I/O-constrained blocks. Place DDR PHY, USB PHY, PCIe PHY along die edges — they need direct pad connections and their bump patterns dictate adjacent placement. DDR PHY south edge (closest to board DDR routing), USB/PCIe west edge.

2. CPU–SRAM adjacency (highest priority):
The 32× SRAM banks should form an L-shape around the CPU cluster. Target ≤300µm wire length from CPU cache controller to SRAM array. This is your #1 timing-critical interface at 3GHz+.

3. Power domains:
• Always-on: PMU, boot ROM, always-on SRAM (retention)
• CPU_PD: switchable, level-shifters on all CPU→fabric interfaces
• GPU_PD, NPU_PD: independent gating, share the same isolation cell library
• Never put a power domain boundary across a timing-critical path

Best Practice
How to Get the Best Results

Claude is most powerful when you treat it like a knowledgeable colleague — give it full context, specify the tool chain, and ask for explanations alongside code.

01

Set the role and context first

Open every technical session with a role primer — it shifts Claude into the right reasoning mode.

"You are a senior RTL design engineer with 10 years at a tier-1 semiconductor company. Target tool: Synopsys DC. Process: TSMC 7nm FF."
02

Paste the actual artifact — not a summary

Paste the full Verilog module, the exact timing report section, or the precise lint warning message. Claude analyzes what's there, not what you describe.

"Here is the full module — lines 1–240 — please review for unintended latches: [paste code]"
03

Specify constraints explicitly

Non-blocking assignments, target tool, technology node, clock frequency — state them in the prompt, not as assumptions.

"Use only non-blocking assignments in sequential blocks. Target Xilinx UltraScale+. Synthesize at 250MHz."
04

Ask for explanation alongside code

Claude generates better RTL when forced to explain its choices — the explanation reveals whether it understood the spec.

"Write the Verilog and then explain why you chose this synchronizer topology over a 2-phase handshake."
05

Always verify synthesized output

Run Claude's code through your linter and synthesizer before committing. Claude is a productivity multiplier, not a rubber stamp — your DV flow is the ground truth.

Lint with Spyglass → Simulate with VCS/Questa → Synthesize with DC → close with PT
Pro Tips
Power-User Prompting for VLSI
T1

Use "Projects" for persistent context

In Claude.ai, create a Project and paste your design spec, naming conventions, and tool constraints into the System prompt — every conversation inherits it.

T2

Iterative refinement beats one-shot

Start with the module skeleton, then ask Claude to add reset logic, then add assertions, then write the testbench. Each step builds on verified output.

T3

Ask it to find its own bugs

After generating RTL, ask: "Review this code you just wrote for CDC issues, incomplete case statements, and blocking assignment misuse." Claude catches ~80% of its own errors.

T4

Paste error messages verbatim

Copy the exact Vivado, DC, or Questa error including the line number. Claude uses the error code and message together — paraphrasing loses critical detail.

T5

Use it as a code reviewer, not just generator

Paste your existing RTL and ask: "Review this for synthesis-unfriendly patterns, latch inference risk, and any protocol violations against AXI4 spec section 3.1."

T6

Request multiple alternatives

Ask: "Give me 3 implementations of this arbiter — round-robin, priority, and weighted — with a tradeoff table. I'll pick based on area budget." Claude is best when comparing options.

FAQ
Common Questions
How can Claude help RTL design engineers?

Claude generates synthesizable Verilog/SystemVerilog from specs, writes FSMs, reviews CDC issues, explains synthesis warnings, and produces testbenches. It handles parameterized designs, AXI/APB interfaces, and standard cell-aware coding patterns.

Can Claude write SystemVerilog assertions?

Yes — Claude generates concurrent SVA properties, sequences, and cover points from plain English. Describe the protocol behavior and it produces the assertion. Always simulate to confirm the assertion fires correctly on both passing and failing cases.

Is Claude useful for STA?

Claude interprets Primetime and Tempus reports, explains setup/hold violations, writes SDC constraints, debugs false path exceptions, and explains OCV/CPPR concepts. Paste the exact timing path report for best results.

Is it safe to paste proprietary RTL into Claude?

Review your company's IP policy first. For proprietary designs, use Claude via API with data privacy agreements, or use anonymized/abstracted versions of the code for review. claude.ai Pro/Team plans have stronger data handling commitments — check Anthropic's terms for your organization.

What are the best prompting practices for VLSI?

Always specify the target tool (Vivado, DC, Innovus), technology node, clock frequency, and exact constraints. Use role-priming. Paste full artifacts — not summaries. Ask for explanations alongside code. Verify all generated RTL through your lint/simulation/synthesis flow.
